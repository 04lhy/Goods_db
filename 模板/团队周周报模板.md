#### 硬件配置优化：
*   **CPU：** 64 位、高主频、高缓存，高并行处理能力 E5。
*   **内存：** 大内存、大内存位宽，尽量不要用 SWAP。
*   **硬盘：** 15000RPM、RAID5、raid10，使用硬件阵列卡，或使用 SSD。
*   **网络：** 标配的千兆网卡，mysql 服务器尽可能和使用它的 web 服务器在同一局域网内，尽量避免诸如防火墙策略等不必要的开销，双网线提供冗余。

#### 操作系统的优化：
1. IO 优化。
2. 创建一个磁盘分区格式化成 xfs，专门存放 data。
3. 网卡的优化——网卡可以采用 Bonding 技术，实现网卡负载均衡高可用。

#### 数据库设计和规划：
1. **架构设计方面：** 纵向拆解；横向拆解。
2. **存储引擎的选择：**
    *   **MyISAM：** 数据库并发不大，读多写少，而且都能很好的用到索引，sql 语句比较简单的应用，TB 数据仓库。
    *   **InnoDB：** 并发访问大，写操作比较多，有外键、事务等需求的应用，系统内存较大。
3. **命名规则：**
    *   多数开发语言命名规则：比如 MyAdress。
    *   多数开源思想命名规则：比如 my_address。
    *   避免随便命名，符合目录名的规则。
4. **索引：**
    *   **优点：** 极大地加速了查询，减少扫描和锁定的数据行数。
    *   **缺点：** 占用磁盘空间，减慢了数据更新速度，增加了磁盘 IO。
    *   **添加索引原则：**
        *   选择唯一性索引。
        *   为经常需要排序、分组和联合操作的字段建立索引。
        *   为常作为查询条件的字段建立索引。
        *   限制索引的数据，索引不是越多越好。
        *   尽量使用数据量少的索引，对于大字段可以考虑前缀索引。
        *   删除不再使用或者很少使用的索引。
        *   结合核心 SQL 优先考虑覆盖索引。
        *   忌用字符串做主键。

#### MySQL 配置优化：
1. **对查询进行缓存：** 优化原则是给 mysql 的资源要适度，太少施展不开，太多会拖累 OS。
2. **启用表缓存：** mysqld 对打开的文件进行缓存以加快读取。
3. **启用关键字缓冲区：** 增加索引缓冲区大小以提升处理速度。
4. **mysql 资源链接优化。**

#### SQL 查询优化：
1. 在 InnoDB 中统计行数建议另弄统计表。
2. 尽量不要在数据库中做运算。
3. 避免负向查询和 % 前缀模糊查询。
4. 不在索引列做运算或者使用函数。
5. 生产环境不使用 `select *`。
6. 使用 limit 减少返回行数。
7. avoid 隐式类型转换（字符型用引号，数字型不用）。
8. SQL 关键词用大写，避免重复编译。
9. 联表查询遵循小结果集驱动大结果集。
10. 开启慢查询，定期用 explain 优化。

### Date：2021.7.8 星期四

**学习计划：**
阅读文档，了解 9 个改进方向，选定 2 个大体优化方向，并尝试小组成员分任务。

**学习总结：**

**1. 网络连接和网络通信协议子系统**
*   **通信过程：** 请求 -> 检查缓存 -> SQL 解析/预处理 -> 执行计划 -> 调用引擎 API -> 返回结果。
*   **连接方式：**
    *   **Unix 套接字：** 仅限 Linux/Unix 同机通信。
    *   **命名管道和内存共享：** Windows 同机通信。
    *   **TCP/IP 套接字：** 跨主机通信，使用最广。

**2. 数据库备份子系统**
*   **mysqldump 备份过程：** flush tables -> flush table with read lock (只读状态) -> 设置 RR 隔离级别 -> 开启事务 -> 获取 GTID/binlog 偏移量 -> 解锁 -> 导出数据/结构 -> 回滚/释放断点。

**3. 日志子系统**
*   **redo log：** InnoDB 特有，WAL 技术（先写日志再写磁盘），支持 crash-safe，固定大小循环写入。
*   **binlog：** Server 层日志，所有引擎通用，用于归档。

**4. 查询解析和查询优化**
*   学习 `parse_sql()` 过程。使用 **bison (yacc 开源版)** 作为语法解析器。在 lex 词法解析阶段解析出 select, from, where 等 token，再由 `sql_yacc.cc` 根据 BNF 规则解析出语法树。
