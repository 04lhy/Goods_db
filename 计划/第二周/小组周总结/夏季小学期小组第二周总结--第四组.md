**刘晗阳（组长）**牵头全组集中修复 Phase 1 遗留问题：

- **Bug 1（rnd_pos 返回 false）**：通过 GDB 和 fprintf 逐层追踪 `rnd_pos → GetTuple → FetchPage → GetTuple` 调用链，发现 `TableHeap::GetTuple` 内部全部成功（FetchPage OK、slot 有效、数据完整），最终定位为**测试断言 Bug**：`rnd_pos` 返回 0 表示成功（C 惯例，与 `rnd_init`/`rnd_next` 一致），但测试使用 `ASSERT_TRUE(handler.rnd_pos(...))` 判断——`ASSERT_TRUE(0)` 在 C++ 中为 false。修复：将 `ASSERT_TRUE` 改为 `ASSERT_EQ(..., 0)`。

- **Bug 2（UpdateAndDelete 超时/死锁）**：定位到三个根因——(1) `TableIterator` 缺少析构函数，销毁时不释放 R-latch 和 pin，导致 latch 泄漏；(2) `Count()` 在末尾调用 `Reset()` 重新 R-latch 第一个页面后不释放；(3) 测试在 `rnd_next` 循环内直接调用 `delete_row()`，此时迭代器持有 R-latch，而 `MarkDelete` 需要 W-latch → 死锁。修复：添加 `~TableIterator()` 析构函数、移除 `Count()` 中多余的 `Reset()`、测试改为先收集 RID 再执行写操作。

- **Bug 3（BPlusTreeIndex ASan SIGSEGV）**：由 Bug 2 的 latch/pin 泄漏连带导致，Bug 2 修复后 ASan 测试全部通过，零内存错误。

同时推动修复三个架构问题：
- **问题 5（Scan+Write 同页死锁）**：测试层面已修复，UpdateExecutor/DeleteExecutor 遵循"先收集 RID → 释放扫描锁 → 执行写操作"模式。

## 存在的问题与难点

1. **B+Tree 删除的借位与合并未完整实现**：当前 `HandleUnderflow` 仅调用 `AdjustRoot` 收缩根节点，未实现兄弟页借位（rotate）和页合并（merge）。极端删除场景下空间利用率可能低于 50%。刘晗阳已列入第三周优化计划。

2. **Hash Index 的桶合并与目录收缩未实现**：删除仅从桶页移除 KV，目录膨胀后不自动收缩。刘晗阳已列入第三周优化计划。

3. **WindowFunctionExecutor 仅支持排名函数**：窗口聚合（SUM/AVG OVER）和帧规范（ROWS BETWEEN）框架已搭建但计算逻辑待完善。刘晗阳已列入第三周补充计划。

4. **外部归并排序未完整实现**：当前 SortExecutor 仅支持内存排序（`std::sort`），当数据量超过 BufferPool 可用内存时缺少外部归并排序降级路径。李锦华已列入第三周。

5. **Parser 内存泄漏（libpg_query 已知问题）**：多测试连续运行时 `pg_parser_cleanup()` 释放全局 MemoryContext 导致 heap-use-after-free。当前通过静态 Parser 单例缓解，长期需升级 libpg_query 或显式管理 MemoryContext。

6. **Lock Manager/Transaction Manager 为简化实现**：仅提供基础的行级锁和 Begin/Commit/Abort 接口，缺少完整的死锁检测、MVCC 多版本链、WAL 日志和 Recovery 机制。这是第三至四周的持续工作。

7. **Optimizer 列裁剪规则为简化实现**：规则框架已搭建但实际列裁剪逻辑标记为 no-op（需要 Schema 分析支持），不影响查询正确性但可能多传输不必要的列数据。

8. **命令行 Shell 工具未实现**：linenoise 集成和 goods_db_shell 未完成（第三阶段任务）。
