# goods_db 实验整体计划

## 项目定位

基于 BusTub 技术架构，构建面向**新零售多仓储电商物流场景**的关系型数据库系统 **goods_db**，满足《数据库系统设计》课程 9 大设计模块要求。

---

## BusTub 技术 → 十个模块 对照表

| 设计模块 | BusTub 可直接复用的组件 | 需要自行开发的部分 |
|---------|----------------------|------------------|
| 设计一：存储引擎 | `storage/` 全部（DiskManager, BufferPoolManager, TableHeap, Tuple, Page, B+Tree, HashIndex, Replacer） | 仓库（warehouse）/ 库存批次（inventory lot）/ 物流单表布局 |
| 设计二：引擎接口 | Executor 工厂模式参考 | **handler 类 + handlerton 单态结构**（参照 MySQL 设计） |
| 设计三：数据库管理 | `bustub_instance.cpp`、`tools/shell/` | goods_db_admin 管理工具（状态、线程、Flush、Reload） |
| 设计四：备份 | 无 | **全量/增量 SQL dump 工具**（遍历 catalog → 生成 INSERT 脚本） |
| 设计五：日志 | `recovery/log_manager.cpp`（WAL 基础） | Error 日志 / Query 日志 / **Binary Log + goods_db_binlog 工具** |
| 设计六：网络协议 | `tools/shell/` 基础网络 | TCP/IP 协议栈 + Protocol 类层次（Binary/Text/Local） |
| 设计七：线程内存 | BufferPool 并发、Transaction/Lock Manager | 连接线程池、内存分配器、启动流程（mysqld 风格 main） |
| 设计八：查询优化 | `binder/` + `planner/` + `optimizer/` + `execution/` 全套 | 新零售场景 SQL 扩展（仓库 JOIN、物流状态查询等） |
| 设计九：安全管理 | 无 | 用户认证（check_connection）+ 权限控制（check_access）+ ACL 表 |
| 设计十：桌面客户端 | 无 | **goods_db_studio（Qt 6 Widgets）**——SQL 编辑器 / 结果浏览器 / 对象树 / 管理面板 / 备份向导 / 日志查看器 |

## 分阶段实施计划

### 第一阶段：核心存储层（第 1-4 周）

> 覆盖模块：设计一（存储引擎）+ 设计二（引擎接口）

#### 1.1 项目骨架搭建
- [ ] 在 BusTub 同级创建 `goods_db/` 工程目录
- [ ] 复用 `cpp-dev:24.04` Docker 镜像
- [ ] 搭建 CMake 构建体系（参考 BusTub 的 CMakeLists.txt）
- [ ] 引入 BusTub 的 `third_party/`（fmt, googletest, argparse, linenoise 等）

#### 1.2 存储引擎核心（设计一）★★★ 最重要

```
直接复用/改造 BusTub 技术：
├── DiskManager         → 磁盘页管理（扩展多文件支持）
├── BufferPoolManager   → 缓冲池（LRU-K / ARC 替换策略）
├── TablePage / Tuple   → 页面布局 + 行级存储
├── TableHeap           → 表堆组织（Insert/Update/Delete/SeqScan）
├── BPlusTreeIndex      → B+ 树索引（随机读写 + 区间查询）
├── HashTableIndex      → 哈希索引（点查优化）
└── PageGuard           → RAII 风格的页面锁保护
```

**新零售扩展：**
- [ ] 设计 warehouse（仓库）、inventory（库存）、shipment（物流）表的数据页布局
- [ ] 实现 DDL 建表逻辑（Create/Open/Close/Delete 表）
- [ ] 实现行级 CRUD（insert_tuple / update_tuple / delete_tuple / get_tuple）

#### 1.3 存储引擎接口层（设计二）★★★

```
参照 MySQL handler/handlerton：
├── class handlerton        → 单态模式，管理存储引擎实例
├── class handler           → 表操作抽象基类
│   ├── open() / close()   → 表打开/关闭
│   ├── rnd_init() / rnd_next() / rnd_pos()  → 全表扫描
│   ├── index_read() / index_next()           → 索引扫描
│   ├── write_row() / update_row() / delete_row()
│   └── create() / delete_table()
└── handler 工厂            → 根据引擎名创建具体 handler 实例
```

- [ ] 定义 `sql/handler.h`、`sql/handler.cc`、`sql/handlerton.h`
- [ ] 实现 handlerton 单态注册机制
- [ ] 封装 BusTub 的 TableHeap + BPlusTree 为 handler 接口
- [ ] 支持引擎插件化：运行时注册/切换存储引擎

### 第二阶段：查询引擎层（第 5-8 周）

> 覆盖模块：设计八（查询解析 + 查询优化）

#### 2.1 查询解析（设计八 前半）★★★

```
直接复用/改造 BusTub：
├── binder/        → SQL 语句绑定（解析 AST → BoundStatement）
├── libpg_query    → PostgreSQL 语法解析器（第三方库，已包含）
├── planner/       → 逻辑计划生成（BoundStatement → AbstractPlanNode）
└── optimizer/     → 查询优化器
    ├── 谓词下推 (filter → scan)
    ├── NLJ → HashJoin 改写
    ├── NLJ → IndexJoin 改写
    ├── SeqScan → IndexScan 改写
    ├── 列裁剪 (column pruning)
    └── Sort+Limit → TopN 优化
```

**新零售 SQL 扩展：**
- [ ] 聚合函数（SUM/AVG/COUNT/MAX/MIN）→ `aggregation_executor`
- [ ] 多表 JOIN（HashJoin / NestedLoopJoin / IndexJoin）
- [ ] 窗口函数 → `window_function_executor`
- [ ] 外部排序 → `external_merge_sort_executor`

#### 2.2 查询执行器（设计八 后半）

```
BusTub 现有执行器全部复用：
├── seq_scan_executor      → 全表扫描
├── index_scan_executor    → 索引扫描
├── insert/update/delete   → DML 操作
├── filter_executor        → 过滤
├── limit_executor         → LIMIT 子句
├── sort_executor          → 排序
├── aggregation_executor   → 聚合
├── hash_join_executor     → 哈希连接
├── nested_loop_join       → 嵌套循环连接
├── nested_index_join      → 索引嵌套连接
├── projection_executor    → 投影
├── values_executor        → VALUES 子句
└── window_function_executor → 窗口函数
```

- [ ] 实现 Volcano 模型迭代器（每个 Executor 的 Init/Next 接口）
- [ ] 扩展为新零售场景：库存汇总查询、物流状态追踪查询

### 第三阶段：系统服务层（第 9-13 周）

> 覆盖模块：设计五（日志）+ 设计六（网络协议）+ 设计七（线程/内存）+ 设计九（安全管理）

#### 3.1 网络通信协议（设计六）★★

- [ ] 基于 TCP/IP 实现网络通信层
  ```
  ├── Protocol (抽象基类)
  │   ├── Protocol_binary   → 二进制协议（Prepared Statement 用）
  │   ├── Protocol_text     → 文本协议（终端交互用）
  │   └── Protocol_local    → 本地 Unix Socket（可选）
  ├── net_serv.cc           → 网络服务读写/分包
  └── Connection            → 每连接一个 Connection 对象
  ```
- [ ] 参考设计方案：`Protocol::store()` / `Protocol::store_result()` / `Protocol::send_ok()` / `Protocol::send_error()`
- [ ] 支持多客户端并发连接

#### 3.2 线程与内存管理（设计七）★★

- [ ] 线程池管理（每个客户端连接 → 一个处理线程）
  ```
  ├── ThreadPool            → 预创建线程 + 任务队列
  ├── ConnectionHandler     → 每连接生命周期管理
  └── main() in goods_db.cc → 启动流程
       ├── 解析配置
       ├── 初始化存储引擎
       ├── 启动网络监听
       └── 进入主循环
  ```
- [ ] 内存分配管理（Buffer Pool 大小可配）
- [ ] 并发控制：Lock Manager（2PL）+ Transaction Manager（MVCC 参考）

#### 3.3 日志子系统（设计五）★★

```
├── Error Log              → 服务器错误/警告（启动失败、崩溃信息）
├── Query Log              → 记录所有 SQL 查询（支持按用户/时间过滤）
├── Binary Log (核心)       → WAL 机制扩展
│   ├── binlog 事件格式    → Query_event / Row_event / Xid_event
│   ├── binlog 写入        → 事务提交前刷盘
│   └── binlog 轮转        → 文件大小限制，自动切换
└── goods_db_binlog 工具   → 解析/查看二进制日志内容
```

- [ ] BusTub 的 `LogManager`（WAL）可直接作为 Binary Log 的基础
- [ ] 扩展 Error Log 和 Query Log
- [ ] 实现 mysqlbinlog 风格的 binlog 解析工具

#### 3.4 安全管理（设计九）★

- [ ] 用户认证系统
  ```
  ├── goods_db.user 系统表   → 用户名/密码/主机
  ├── check_connection()    → 连接时身份验证
  ├── check_access()        → 操作权限校验
  └── ACL 权限粒度          → 数据库级 / 表级 / 列级
  ```
- [ ] 密码哈希存储（SHA-256）
- [ ] GRANT / REVOKE 命令支持


#### 4.2 备份子系统（设计四）★★

## 开发环境使用

```bash
# 1. 构建通用 C++ 开发镜像（只需一次）
cd /home/lhy/桌面/Goods_db
docker compose build

# 2. 进入 BusTub 环境学习源码
docker compose run --rm bustub bash

# 3. 后续创建 goods_db 工程时，用同一镜像创建新容器
docker run -it --rm -v "$(pwd)/goods_db:/workspace" cpp-dev:24.04 bash

# 4. 编译桌面客户端（需 Qt 6.5+）
cd goods_db_studio/build
cmake .. -DCMAKE_BUILD_TYPE=Debug
make -j$(nproc)
./goods_db_studio
```
