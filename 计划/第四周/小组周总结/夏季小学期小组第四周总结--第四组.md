**学习计划：**

**学习总结：**

今天是深度 Bug 修复日——IS NULL 永远返回 FALSE 的根因定位耗费大量调试时间，最终发现是一行代码的执行顺序问题，影响范围覆盖所有含 NULL 列的表。

**刘晗阳（组长）**上午定位 IS NULL / IS NOT NULL 永远返回 FALSE 的根因。排查过程：在 FilterExecutor、BoundComparison、BoundUnaryOp、CreateFromValues、GetValue 逐层添加 `std::cerr` 调试输出 → 发现 GetValue 正确返回 NULL（null bitmap=0x10）→ BoundUnaryOp::Evaluate 从未进入 IS_NULL 分支 → 审查代码发现 `BoundUnaryOp::Evaluate` 中 NULL 传播检查 `if (val.GetTypeId() == TypeId::INVALID) return Value()` 在 `IS_NULL` 检查之前执行，当列值为 NULL 时函数提前返回 INVALID。修复方案仅移动 3 行代码（将 IS_NULL/IS_NOT_NULL 检查移到 NULL 传播之前），因为 IS_NULL 的语义本身就是"检测 NULL"，NULL 输入对其是合法且预期的。影响范围：所有含 NULL 列的表（warehouses.status、inventory.last_updated、shipments.arrive_date、customers.address/phone 等），DELETE/UPDATE/SELECT WHERE IS NULL 均不生效。

下午修复内存泄漏——`goods_db_main.cpp:95` 中 `new handlerton()` 分配的内存在程序退出时未释放，LeakSanitizer 检测到 56 字节泄漏。改为 `static` 局部变量，生命周期跟随进程。验证权限隔离功能——确认 AuthManager + ACL 权限系统完整可用：默认三级用户体系（root 全部权限/空密码、admin 全部权限/密码 12345、guest 仅 SELECT/空密码），guest 用户执行 DELETE 操作被正确拒绝。创建 `启动指令.md` 文档——编译、启动、加载演示数据的一站式命令速查。与肖清峰联调集成测试，排查 warehouses 表删除/编辑失败问题，验证 IS NULL 修复后全表操作正常。

### 设计二：存储引擎接口子系统 ✅ 全部完成

### 设计六：网络连接和通信协议 ✅ 全部完成

- 完整 Protocol 类层次：`Protocol` 抽象基类 → `Protocol_text`（文本协议，支持 telnet/nc 调试）+ `Protocol_binary`（二进制协议）
- `Connection` 类：封装 POSIX socket，阻塞读取 4 字节 header + payload（最大 16MB），6 状态机（INIT/AUTH/READY/QUERYING/SENDING/CLOSING）+ 全局连接统计
- `SocketServer`：TCP 监听（bind + listen + accept，backlog=128）
- `GoodsDbProtocol` 客户端：Qt-free TCP 客户端（非阻塞 connect + select 超时），支持 Connect/Authenticate/Execute/Ping/Disconnect 全流程

### 设计七：线程和内存分配子系统 ✅ 全部完成

- `ThreadPool`：预创建 N 个 worker 线程（N 可配置，默认 CPU 核心数），线程安全任务队列（std::mutex + std::condition_variable），`Submit(task)` 返回 `std::future<Result>`，支持优雅关闭 `Shutdown()` 和强制关闭 `ShutdownNow()`

- `main()` 6 阶段启动流程：解析命令行参数 → 初始化存储层 → 注册存储引擎 → 初始化 AuthManager/LogManager/LockManager/TransactionManager → 创建 ExecutionEngine → 启动 ThreadPool + Listen
- 内存泄漏修复：`handlerton` 分配从 `new` 改为 `static`，LeakSanitizer 验证零泄漏
